# Веб-интерфейс для отображения рыночных сделок (okx_data_set_record_data_2)

## 1. Цель

Веб-интерфейс в стиле TradingView для просмотра последних рыночных данных из таблицы `okx_data_set_record_data_2`: свечной график, переключение масштабов/уровней теории Доу, панель объёмов с разбивкой buy/sell.

---

## 2. Источник данных

- **Таблица:** `okx_data_set_record_data_2`
- **Структура:** каждая строка — один **бар** (агрегат по сделкам), а не отдельная сделка.
- **Ключ:** `(symbol_id, start_trade_id)`.
- **Поля (релевантные для GUI):**
  - Идентификация: `symbol_id`, `start_trade_id`, `end_trade_id`
  - Время: `start_timestamp_ms`, `end_timestamp_ms`
  - OHLC: `open_price`, `high_price`, `low_price`, `close_price`
  - Объём: `total_volume`, `buy_volume` (sell_volume = total_volume - buy_volume)
  - Количество: `total_quantity`, `buy_quantity`, `total_trades_count`, `buy_trades_count`

**Вычисляемые поля (на бэкенде при формировании ответа API):**
- `buy_volume_percent` = `buy_volume / total_volume` ∈ [0, 1]
- `sell_volume_percent` = `1 - buy_volume_percent` ∈ [0, 1]
- `total_volume_log2` = `log2(total_volume)` — для высоты бара объёма (логарифмическая шкала). Формула как в расчётном датасете: `pl.col('total_volume').log(base=2).alias('total_volume_log2')`, референс: `C:\Repositories\trading_bot_2\src\data\calculation_module.py` (блок «total_volume_log2»).

Существующий код чтения: `main/process_data/data_set_record_fetcher.py` (Polars + connectorx), схема: `main/save_final_data_set_2/schemas.py`.

**Символы:** список брать из enum `SymbolId` (сейчас: BTC_USDT, ETH_USDT). В UI — выбор символа в первой версии.

| Параметр | Описание | По умолчанию | Пример env |

## 4. Масштабы (агрегация баров)

- **x1** — без агрегации: один бар БД = одна свеча на графике (по умолчанию).
- **x2, x4, x8, …, x256, x512, x1024, x2048** — объединение подряд идущих баров в один:
  - **Правила агрегации** (как в `trading_bot_2`, `CandleAggregator`):
    - `open_price` — first, `close_price` — last
    - `high_price` — max, `low_price` — min
    - `total_volume`, `buy_volume` — sum
    - `total_quantity`, `buy_quantity`, `total_trades_count`, `buy_trades_count` — sum
    - Времена: `start_timestamp_ms` — first, `end_timestamp_ms` — last; `start_trade_id` — first, `end_trade_id` — last
  - Группировка: по индексу строки `index // N`, где N — множитель масштаба (2, 4, 8, …).

Референс: `C:\Repositories\trading_bot_2\src\data\aggregation.py`, конфиг масштабов: `configs/model/EnhancedRealTimeGRUModel.yaml` (scale_features: x1…x16384). В данном проекте для GUI достаточно до x2048 (или расширить при необходимости).

В выпадающем списке интерфейса: выбор масштаба (x1, x2, …, x2048) переключает отображаемый ряд свечей. **Агрегация выполняется на бэкенде** — фронт получает уже готовые агрегированные свечи.

## 5. Уровни по теории Доу (final_tensors, уровни 1–5)

**Решение для первой версии:** в выпадающем списке — масштабы (x1…x2048) и пункты «Уровень 1»…«Уровень 5» как **заглушки (TODO)**. При выборе уровня — показывать сообщение «В разработке» или оставить переключение без данных до реализации.

**Референс для будущей реализации:** в репозитории **trading_bot**:
- `src/data/dow_state.py` — калькуляторы трендов (внутри states). Класс `DowState(sequence_length)`; метод `add_row(row)` — «скармливает» очередную строку датасета всем калькуляторам; метод `get_tensors()` — возвращает финальные тензоры по уровням (например, `final_tensors_level_1`, `final_tensors_level_2`, …).
- `src/data/dataset.py` — использование: создаётся `DowState`, в цикле по строкам вызывается `add_row`, после прогона всех строк вызывается `get_tensors()` для получения итоговых тензоров.

По сути: каждую строку из датасета передаём в калькуляторы, после обработки всех строк — получаем финальные тензоры. В **okx_data_saver** при реализации можно портировать/подключить эту логику или отдельный сервис.

## 6. Объёмная панель (внизу графика)

- Один вертикальный бар на каждую свечу (в выбранном масштабе).
- **Высота бара** — по логарифмическому объёму: от `total_volume_log2` (например, нормализация по min/max по видимой области или глобально).
- **Состав бара:**
  - **Нижняя часть** — зелёная: доля объёма на покупку (`buy_volume_percent`), если не нулевая.
  - **Верхняя часть** — красная: доля объёма на продажу (`sell_volume_percent`), если не нулевая.
- Итого: один бар = зелёный сегмент [0, buy_volume_percent] + красный сегмент [buy_volume_percent, 1] по высоте, масштаб высоты — от log2(total_volume).

## 7. Интерфейс в стиле TradingView

- **График:** свечи (OHLC) по времени (или по trade_id), синхронная прокрутка/масштабирование с панелью объёмов.
- **Тулбар:** выбор символа (symbol_id), выбор масштаба/уровня (выпадающий список: x1, x2, …, x2048 и при наличии — Уровень 1…5).
- **Автообновление:** раз в N секунд (параметр конфига) подгрузка последних данных без сброса позиции графика (или с опцией «прижать к правому краю»).
- **Технологии:** на усмотрение реализации (например, FastAPI + React/Vue + lightweight-charts или другой чарт-библиотека; можно начать с простого HTML/JS + Fetch для «слона по кусочкам»).

### REST vs WebSockets

**Рекомендация: REST + периодический опрос (polling).**

- **Первая загрузка:** один GET-запрос с параметрами `symbol_id`, `limit`, `scale`. При большом лимите (до 1M баров) ответ может быть тяжёлым — имеет смысл сразу предусмотреть **пагинацию** (например, `limit` + `offset`) или **потоковую отдачу** (chunked transfer), чтобы не держать в памяти один гигантский JSON и не блокировать браузер.
- **Автообновление раз в 30 сек:** клиент раз в N секунд повторяет тот же GET (или GET с «хвостом» — только свежие бары после последнего `end_trade_id`). Просто, без отдельного канала.
- **WebSockets** дают серверный push и меньше лишних запросов при частом обновлении, но усложняют бэкенд и деплой. Их можно добавить позже для режима «живой ленты» (сервер пушит новые бары по мере появления), оставив REST для начальной загрузки и опционального polling.

Итого: старт с **REST** (в т.ч. с пагинацией/streaming для большого лимита) и **polling** для автообновления; WebSocket — при необходимости в следующих итерациях.

## 9. План реализации (по шагам)

Разбиение на этапы, чтобы «есть слона по кусочкам»:

1. **Конфиг и среда**  
   Добавить в настройки (config/env): лимит записей, интервал обновления (сек), опционально список масштабов. Подключить к будущему веб-серверу.

2. **API для данных**  
   REST: эндпоинт(ы) для получения последних N баров по `symbol_id`. Ответ: JSON с полями OHLC, временем, buy_volume_percent, sell_volume_percent, total_volume_log2. Query-параметры: `symbol_id`, `limit`, `scale` (x1, x2, …), при необходимости `offset` для пагинации или потоковая отдача (chunked) при большом лимите.

3. **Агрегация масштабов на бэкенде**  
   Реализовать агрегацию баров (алгоритм как в CandleAggregator): при запросе с `scale=xN` отдавать уже объединённые свечи.

4. **Фронт: скелет и чарт**  
   Минимальная страница: загрузка данных по API, отображение свечей (и при желании объёмов) с выбором масштаба в выпадающем списке.

5. **Фронт: объёмная панель**  
   Бары объёма по total_volume_log2, зелёный/красный по buy_volume_percent / sell_volume_percent.

6. **Автообновление**  
   Таймер на фронте (или периодический запрос) каждые N секунд, без сброса видимой области (или с опцией «держать хвост в поле зрения»).

7. **Теория Доу (опционально)**  
   Либо заглушки в выпадающем списке, либо интеграция с расчётом трендов (отдельный сервис/модуль) и отображение уровней 1–5.

8. **Улучшения UX**  
   TradingView-like: курсор, крест, зум, сжатие/растяжение по осям, легенда, настройки темы.
